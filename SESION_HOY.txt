TEMA PRINCIPAL: Idempotencia en el envio de emails del sistema
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Que es la idempotencia?
  Que llamar la misma operacion N veces produce el mismo resultado
  que llamarla una sola vez. Clave cuando hay reintentos de red.

Problema que resolvimos:
  identity llama a notifications para enviar un email. Si la red
  falla y identity reintenta, notifications enviaba el email DOS
  veces (verificacion duplicada, reset duplicado, etc.).

1. services/identity/src/adapters/http/auth-routes.ts
   sendInternalNotification ahora envia el header:
     x-idempotency-key: <clave deterministica basada en el token hash>

   Claves por operacion:
     verify-email        -> "verify-email:{verificationHash}"
     forgot-password     -> "forgot-password:{tokenHash}"
     resend-verification -> "resend-verify:{tokenHash}"

   Como el hash del token no cambia entre reintentos, la misma
   operacion siempre genera la misma clave -> el servidor puede
   detectar el duplicado.

2. services/notifications/src/adapters/http/business-routes.ts
   Agregado el IdempotencyStore (Map en memoria con TTL de 24h):

     _idempotencyStore: Map<key, expiryTimestamp>

   Dos funciones exportadas:
     isDuplicate(key)   -> true si la clave ya fue procesada y sigue vigente
     markProcessed(key) -> registra la clave con TTL de 24 horas

   Comportamiento del handler POST /notifications/send-email (sistema):
     Primera llamada con clave  -> envia email + registra clave
                                   -> { sent: true }
     Reintento con misma clave  -> NO envia, responde inmediatamente
                                   -> { sent: true, deduplicated: true }
     Sin clave                  -> siempre envia (comportamiento anterior)
     Clave expirada (>24h)      -> se trata como nueva, envia normalmente

   Compactacion automatica cada 100 escrituras para evitar memory leaks.
