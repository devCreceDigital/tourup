── P0 — Fundamentos ────────────────────────────────────
• Auth frontend en Next.js (App Router + React Context): login, register, logout
• HttpError hierarchy: BadRequestError(400), UnauthorizedError(401),
  ForbiddenError(403), NotFoundError(404), ConflictError(409), ValidationError(422)
• JWT middleware en el API Gateway con jose (HS256)
• Hashing de contraseñas con scrypt (formato scrypt$v1$<salt>$<hash>)
• InMemoryRepository pattern para tests unitarios

── P1 — Resiliencia y observabilidad ───────────────────
• Logger estructurado JSON (packages/service-runtime/src/logger.ts)
    - info/debug -> stdout | warn/error -> stderr
    - Nivel controlado por LOG_LEVEL env var
• Outbox resiliente:
    - Contador de reintentos por evento (retryCount)
    - Máximo de reintentos desde OUTBOX_MAX_RETRIES env var
    - lastError truncado a 1000 chars, lastErrorAt registrado
    - log.warn si reintentará, log.error si agotó reintentos
• Health check endpoint /health en los 15 servicios
    - createDbHealthCheck(prisma) pingea DB con SELECT 1
    - Cuenta outboxBacklog y outboxDead
    - Retorna status: "ok" | "degraded" -> 503 si degraded
• Password reset flow:
    - POST /identity/auth/forgot-password (siempre 200, no revela si email existe)
    - POST /identity/auth/reset-password (valida token SHA-256, TTL 1h)
    - Al resetear: revoca todos los refresh tokens activos del usuario

── P2 — Validación y rate limiting ─────────────────────
• Zod v4 centralizado (packages/shared-kernel/src/validate.ts):
    - Schemas: uuidSchema, emailSchema, paginationSchema, currencyAmountSchema,
      currencyCodeSchema, isoDatetimeSchema, phoneSchema, userRoleSchema
    - validate() lanza ValidationError(422) con todos los errores concatenados
    - validatePartial() para PATCHes parciales
• Rate limiter pluggable (apps/api-gateway/src/rate-limit.ts):
    - InMemoryRateLimiterStorage — default, sin dependencias nuevas
    - UpstashRateLimiterStorage — Redis distribuido via fetch REST API,
      se activa si UPSTASH_REDIS_REST_URL + UPSTASH_REDIS_REST_TOKEN están definidos
    - Límites configurables por env:
        RATE_LIMIT_ANONYMOUS_RPM (default 60 req/min por IP)
        RATE_LIMIT_USER_RPM      (default 300 req/min por usuario)
        RATE_LIMIT_USER_RPH      (default 3000 req/hora por usuario)
    - Retorna 429 con header retry-after
• Email verification flow:
    - POST /identity/auth/verify-email
    - POST /identity/auth/resend-verification
    - Controlado por REQUIRE_EMAIL_VERIFICATION=true/false
    - Al registrar sin verificación: emailVerifiedAt se setea en el momento
• Outbox con campos nuevos en los 15 schemas de Prisma:
    - retryCount Int @default(0)
    - lastError  String?
    - lastErrorAt DateTime?
    - @@index([retryCount])

── Idempotencia ─────────────────────────────────────────
✅ sendInternalNotification en identity ahora envía x-idempotency-key:
   - verify-email:       "verify-email:{verificationHash}"
   - forgot-password:    "forgot-password:{tokenHash}"
   - resend-verification: "resend-verify:{tokenHash}"
   Claves determinísticas basadas en el hash del token → el reintento
   de la misma operación siempre genera la misma clave.

✅ POST /notifications/send-email (flujo sistema) deduplica por clave:
   - _idempotencyStore: Map<key, expiryTimestamp>, TTL=24h
   - Primera llamada: envía email + registra clave → {sent: true}
   - Reintento con misma clave: no envía → {sent: true, deduplicated: true}
   - Sin clave: siempre envía (comportamiento anterior sin cambios)
   - Clave expirada (>24h): se trata como nueva, envía normalmente
   - Compactación automática cada 100 escrituras (evita memory leaks)

✅ Tests de idempotencia (services/notifications/src/tests/idempotency.test.ts):
   13 tests — isDuplicate/markProcessed (6 tests) + handler send-email (7 tests)

  PARA LEVANTAR LOCALMENTE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. Copiar variables de entorno:
     cp .env.example .env
   Editar .env y completar: POSTGRES_DB, POSTGRES_USER, POSTGRES_PASSWORD,
   APP_JWT_SECRET, SERVICE_INTERNAL_SECRET

2. Levantar Postgres:
     docker compose up postgres -d

3. Aplicar migraciones:
     .\infra\scripts\migrate-dev.ps1       (Windows)
     bash infra/scripts/migrate-dev.sh     (Linux/Mac)

4. Regenerar clientes Prisma:
     pnpm prisma:generate

5. Levantar todo:
     docker compose up -d

6. Correr tests unitarios:
     pnpm -r --if-present test:unit

   Para levantar solo identity + notifications (desarrollo):
     docker compose up postgres identity notifications -d
